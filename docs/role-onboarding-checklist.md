# Role Onboarding Checklist

## Purpose
This checklist helps new team members onboard into project roles and supports smooth handoff of responsibilities between team members. Use this template when a new person joins the project or when transitioning role ownership.

## General Onboarding (All Roles)

- [ ] Review [Project Management Overview](octoacme-project-management-overview.md)
- [ ] Review [Roles and Personas](octoacme-roles-and-personas.md) to understand team structure
- [ ] Access granted to project repository and documentation
- [ ] Added to relevant communication channels (Slack, Teams, etc.)
- [ ] Introduced to key stakeholders and team members
- [ ] Review current project charter/one-pager
- [ ] Understand current sprint/iteration goals
- [ ] Review recent retrospective notes and action items

---

## Role-Specific Onboarding

### Developer Onboarding

- [ ] Local development environment setup completed
- [ ] Repository cloned and build successful
- [ ] CI/CD pipeline access and familiarity
- [ ] Code review process and standards reviewed
- [ ] Testing framework and coverage expectations understood
- [ ] Assigned first "good first issue" or pairing session scheduled
- [ ] Access to monitoring/logging tools (if applicable)

### Product Manager Onboarding

- [ ] Review product vision and roadmap
- [ ] Access to product analytics and metrics dashboards
- [ ] Understand current OKRs and success metrics
- [ ] Review backlog prioritization criteria
- [ ] Meet with key customers or review recent customer feedback
- [ ] Access to user research tools and repositories
- [ ] Review competitive landscape and positioning

### Project Manager Onboarding

- [ ] Access to project tracking tools (boards, Jira, etc.)
- [ ] Review current project timeline and milestones
- [ ] Review risk register and mitigation plans
- [ ] Understand escalation paths and stakeholder communication cadence
- [ ] Access to documentation repositories
- [ ] Review recent status reports and meeting notes
- [ ] Shadow existing meetings (standups, planning, reviews)

### UX/UI Designer Onboarding

- [ ] Access to design tools (Figma, Sketch, Adobe XD, etc.)
- [ ] Review design system and component library
- [ ] Understand brand guidelines and accessibility standards
- [ ] Access to user research data and personas
- [ ] Review recent usability test findings
- [ ] Understand design review and handoff process
- [ ] Introduced to key users or customer advocates

### Technical Writer / Documentation Lead Onboarding

- [ ] Access to documentation repositories and CMS
- [ ] Review documentation style guide and standards
- [ ] Understand publishing and release workflow
- [ ] Review upcoming features requiring documentation
- [ ] Access to support ticket system for feedback analysis
- [ ] Introduced to subject matter experts (SMEs)
- [ ] Review documentation metrics and goals

### DevOps / Platform Engineer Onboarding

- [ ] Access to cloud infrastructure and IaC repositories
- [ ] Review CI/CD pipeline architecture and configuration
- [ ] Access to monitoring, logging, and alerting systems
- [ ] Review incident response procedures and runbooks
- [ ] Understand deployment processes and schedules
- [ ] Access to secrets management and security tools
- [ ] Review infrastructure documentation and diagrams
- [ ] Shadowing on-call rotation (if applicable)

### Security Lead / Champion Onboarding

- [ ] Access to security scanning tools and dashboards
- [ ] Review security policies and compliance requirements
- [ ] Understand threat model and risk assessment process
- [ ] Review recent security audits and findings
- [ ] Access to vulnerability management system
- [ ] Review incident response plan and contacts
- [ ] Understand security review gates in development process
- [ ] Review secure coding guidelines and training materials

### Customer/Support Advocate Onboarding

- [ ] Access to support ticket system and CRM
- [ ] Review customer feedback channels and processes
- [ ] Understand escalation procedures for critical issues
- [ ] Access to customer analytics and satisfaction metrics
- [ ] Review knowledge base and support documentation
- [ ] Introduced to key customer accounts or segments
- [ ] Understand beta program and early access processes
- [ ] Shadow customer calls or support sessions

## Responsibility Handoff Checklist

When transitioning role ownership from one person to another:

### Knowledge Transfer

- [ ] Schedule handoff meetings (recommend at least 2-3 sessions)
- [ ] Review current work in progress and status
- [ ] Transfer ownership of key documents and artifacts
- [ ] Share access credentials and update ownership in systems
- [ ] Document tribal knowledge and unwritten practices
- [ ] Introduce to key contacts and stakeholders

### Active Work Items

- [ ] Review all assigned/in-progress tasks
- [ ] Clarify priorities and deadlines
- [ ] Transfer ownership in project tracking system
- [ ] Update communication plans with new contact info
- [ ] Schedule follow-up check-ins during transition period

### Documentation Updates

- [ ] Update RACI matrices or responsibility assignments
- [ ] Update contact lists and escalation paths
- [ ] Update project documentation with new role owner
- [ ] Update automated notifications and alerts
- [ ] Verify all access and permissions are transferred

### Validation

- [ ] New role owner has successfully completed first key task
- [ ] No critical knowledge gaps identified
- [ ] Stakeholders informed of transition
- [ ] Outgoing role owner available for questions during transition period
- [ ] Retrospective on handoff process (lessons learned)

## Notes

- Adapt this checklist based on project complexity and team size
- Keep this document updated as processes evolve
- Use this as a starting point; add project-specific items as needed
- Consider creating a buddy/mentor system for onboarding support
